Software Development Life Cycle

Waterfall model

Waterfall model is the simplest model Software Development Life Cycle. All the phases of SDLC will function in linear line one after another. When the first phase is finished then only the second phase and start and so on. The sequential nature of model does not allow us to go back and undo or redo the our actions. It does not work smoothly if there are some issues left at the previous step. This model is best suited when developers already have designed and developed similar software in the past and are aware of all its domain. It provides structure to inexperienced staff. Works well when quality is more important then cost or schedule.

Structured Evolutionary Prototyping model

Developers build a prototype during the requirements phase.
Prototype is evaluated by end users.
Users give corrective feedback. Developers further develop the prototype.
When the user is satisfied , the prototype code is brought up to standards needed for a final product.Some of the steps, which are followed by Structured Evolutionary Prototyping Steps :

1)A Preliminary project plan is developed
2) Partial high-level paper based model is created.
3)The model is source for a partial requirements specification
4)A prototype is built with basic and critical attributes
5)The designer builds : The database, User interface and Algorithmic Functions
6) The designer demonstrates the prototype, the user evaluates for problems and suggests improvements.
7) This loop continues until the user is satisfied with the product.

Incremental model

Each release delivers an operational product. Developing high-risk or major functions first. Each release delivers and operational product. Customer or user can give give feedback or respond for each build. Initial product delivery is faster. Total cost of the complete system is high.

Rapid Application development

RAD is a software development methodology that focuses on building applications in a very short amount of time. It is a high speed adaption of the linear sequential model in which rapid development is achieved by using component based construction .It improved productivity with fewer people means lower costs. It minimizes the risk of not achieving business needs and customer satisfaction. It requires sufficient human resources to create the team for development. Everybody must be committed to complete the process in the time frame.

Spiral model

It is a combination of iterative and Waterfall model. It can be seen as if you choose one and SDLC model and combined it with cyclic process (iterative model). Risk assessment expertise is required. Spiral may continue indefinitely. he phases of this model :
In the first phase, this model considers risk, which often goes un noticed by most other models.
In the Second phase, The model starts with constraints of the software at the starting of the iteration.
In the third phase is of the prototyping the software. This phase includes risk management.
In the last phase of the plan of next iteration is prepared. The model is complex.

V-model

The major drawback of waterfall model is we move to the next stage only when the previous one is finished and there was no chance to go back if something is found worng in later stages. V-model provides means of testing of software at each stage in reverse manner.At every stage, test plans and test cases are created to verify and validate the product according to the requirement of that stage.
For example, in requirement gathering stage the test team prepares all the test cases in correspondence to the requirements. Later, when the product is developed and is ready for testing, test cases of this stage verify the software agains t its validity towards requirements at this stage. This makes both verification and validation go in parallel.

Among the many SDLC methodologies, those that fall under the Agile classification are generally better. Among the Agile SDLC methods, I have a few suggestions: First, we should focus on SDLC methodologies that enhance flexibility, collaboration, and speed of software delivery. In this case, I believe Scrum, Feature-Driven Development, and Extreme Programming (XP) are better options compared to others.

Scrum Model

Scrum's iterative and incremental approach breaks the project up into manageable, small units called sprints, which usually span two to four weeks. This makes it possible for teams to consistently deliver functioning product increments, facilitating prompt feedback and modifications. Improved Collaboration: Through sprint planning, daily stand-up meetings, and retrospectives, Scrum places a strong emphasis on cooperation and teamwork. Development work is divided into "Packets".This promotes communication between stakeholders and team members, guaranteeing that everyone is on the same track about goals and achievements. Flexibility: Even at the very end of the development process, Scrum permits requirements to alter. In dynamic workplaces where user needs may change, this flexibility is essential. Emphasis on Deliverables: By placing a strong emphasis on generating a potentially shippable product increment at the end of each sprint, the team is guaranteed to stay committed to providing value to the client.

Feature-Driven Development (FDD)

Feature-Centric Approach: FDD prioritizes the methodical delivery of concrete, functional features. By organizing and controlling tasks according to client needs, this method assists teams in making sure that the most useful features are created first. Clear and Structured Process: FDD offers an organized framework that includes roles, duties, and procedures. Teams that use this structure are better able to stay accountable and focused while remaining adaptable to changes. Frequent work Tracking: FDD places an extreme value on using feature lists and status updates to monitor work on a regular basis. Stakeholders are able to see the development process and make well-informed judgments since there is transparency. Cooperation with Stakeholders: To guarantee that the finished product meets user expectations, FDD pushes cooperation with stakeholders throughout the feature definition and design stages.

Extreme Programming (XP)

Focus on Technical Excellence: XP encourages the use of best practices in software development, including pair programming, continuous integration, and test-driven development (TDD). Reliable software is delivered more quickly thanks to these techniques, which also improve code quality and decrease errors. client Involvement: Throughout the development process, XP promotes ongoing client input and participation. The product's ability to adjust to evolving requirements and satisfy user needs is guaranteed by this close partnership. Regular Releases: XP promotes regular releases of tiny functional increments so that teams may promptly collect feedback and make the required corrections. This quick delivery cycle improves user response.
